# AI Engineering 技术选型决策笔记

> 本文记录 CodeMind 工程在 AI Engineering 层面的技术辨析与选型思考。  
> 假定读者对 Tool、Skill、MCP、LangGraph、harness engineering 等概念有基础了解。  
> 重点在「为什么这么选」，而非「是什么」或「怎么实现」。

---

## 一、Tool 与 Skill 的边界划分

### 决策背景

在设计 LLM 与业务逻辑的交互层时，面临一个核心问题：**哪些能力应该交给模型决策，哪些应该固定在代码里**。

### 判定

**Tool** 是「模型可见、可选、可参数化的确定性执行单元」。它的执行体不再有模型参与，准确率由代码保证。本工程的 `ToolRegistry` 就是这个角色——每个注册函数背后直接调 ORM，不经过 LLM 生成 Cypher。

**Skill**（在 Anthropic 官方定义中）是「组织成文件目录的说明文档 + 可选脚本」，供强自主 agent 动态发现和加载。它的触发和选择本身依赖模型的主动判断。

**两者的关键区别不是功能，而是执行保证的强度：**

- Tool：结果确定，模型负责选择和传参，代码负责执行
- Skill（Anthropic 标准）：模型负责发现、触发、读取、应用，全程依赖模型的判断质量

本工程最终没有实现标准 Skill 格式。实际存在的 `skills/` 目录里的类，**准确说是 PromptStrategy（提示词策略）**——它管理「告诉模型这一步怎么做」的 prompt 和「校验模型输出」的逻辑。命名借用了 Skill，但职责更接近节点行为规格。

## 二、Anthropic Harness Engineering 的核心判断

### 核心思想提炼

Anthropic 在 *Effective harnesses for long-running agents*（2025-11）里提出的不是新的模型能力，而是一个工程约束框架：

> **让 agent 只在受控环境里做决策，每步的输出都有外部检查点。**

具体体现为：
- 控制流写在代码里（而非让模型自由规划）
- 状态外置（进度文件、git history），而非依赖模型记忆
- 每步只推进少量、可验证的工作，结束前留下清晰的交接物
- 失败有对策，不让模型「自己想办法」

### 对本工程的判定

本工程的 LangGraph 图结构本身就是一个 harness：意图识别 → 工具选择 → 执行 → 渲染，所有路由逻辑在代码里，模型只在每个节点完成一个小任务。

**这和 harness engineering 的精神高度吻合**。差距在于节点内部：原始版本里 LLM 的输出直接流入下一步执行，没有校验检查点。改进方向因此是在每个 LLM 节点的输出和执行之间加代码校验层——而不是去引入更多的模型自主性。

## 三、小模型场景的技术选型影响

### 核心命题

> 模型参数量越小，自主决策范围越大，系统越不稳定。

这不是对小模型的否定，而是工程上的客观约束。基于这个判断，所有技术选型都应该朝「减少模型自主决策面」方向优化。

### 具体影响

**Tool 层**：小模型选工具时容易幻觉工具名、遗漏必填参数。解决方案不是写更好的 prompt，而是：
- 按意图子类型预筛工具（全量 15 个 → 相关的 3～5 个），缩小决策空间
- 输出后加代码校验（白名单 + 参数完整性），非法输出由代码处理，不交还给模型

**Cypher 生成**：对小模型风险最高的节点（完全开放生成）。优化策略按优先级：
1. **扩充 ToolRegistry 模板**：把常见查询模式变成注册工具，让更多请求走确定性路径，减少进入 Cypher 生成的频次
2. **语法校验后再执行**：生成不等于执行，代码校验是检查点
3. **携带错误重试**：失败时把错误原因注入下一次 prompt，比盲目重试更有效

**意图分类**：对小模型来说，2 选 1 分类其实风险不大，但失败时的 fallback 方向很重要。原设计的默认值是 `graph_query`（错误方向），会把无关输入推入复杂链路。正确的安全方向是 `other`——失败时选择最简单的终止路径。

### Skill 文档对小模型的价值判断

如果用 Skill（文档形式）引导小模型，有效的格式要素优先级：

1. **anti-examples（反例）** > 规则列表：直接展示错误写法，比抽象规则更能减少小模型发散
2. **输出格式要求放在 prompt 末尾**：小模型对靠近末尾的指令遵循率更高
3. **每次只给相关的上下文片段**：小模型容易被长上下文淹没，不要把 schema 全量注入

但即便 Skill 写得再好，**自然语言约束终究是软约束**。等价于硬编码约束的只有：有限状态机 + 代码校验 + 代码路由。Skill 覆盖的只是「允许模型发挥的薄层」。

## 四、Anthropic Agent Skills 标准的适用性判断

### 标准定义

Anthropic 于 2025 年 10 月正式发布的 Agent Skills 是「文件系统目录 + SKILL.md + 渐进式加载」的标准格式，核心设计是让**模型自主发现、触发、组合**技能。

### 不适用本工程的根本原因

Agent Skills 的所有机制都建立在一个隐含前提上：**模型有足够强的判断力自主决定何时触发哪个 Skill**。

本工程当前的主要场景恰恰相反——使用小模型，且有明确的业务流程（图查询）。两者的设计哲学直接冲突：

| | Agent Skills | 本工程 |
|---|---|---|
| 控制流由谁决定 | 模型自主判断 | 代码（LangGraph 图结构）|
| Skill 如何触发 | 模型扫描 name/description 自主决定 | 代码路由到对应节点 |
| 适合的模型强度 | 强模型（Claude Code 场景）| 小模型（确定性业务场景）|

更直接的说法：LangGraph 的固定拓扑已经是比 Agent Skills 更强的约束，是更好的 harness。引入 Agent Skills 反而会把已经固定的控制流重新交还给模型判断，对小模型是倒退。

### 未来可能适用的场景

如果 CodeMind 未来要演变为**对外开放、允许第三方开发者扩展能力**的平台，Agent Skills 标准的「可组合、可移植」价值才会显现。当前作为内部工具，没有必要承担这个格式的复杂性成本。

## 五、MCP 的定位

MCP 解决的问题和 Tool/Skill 不同：**Tool 是「能做什么」，Skill 是「怎么用 Tool」，MCP 是「用什么协议暴露和调用」**。

本工程的 `ToolRegistry`（name / description / parameters / handler）在结构上已和 MCP tool schema 等价。MCP 对本工程的实际价值有两个方向：

**对外暴露（MCP Server）**：把图查询能力标准化，供外部 agent 或其他系统调用。这是未来可考虑的方向，当前工程结构已为此铺路。

**对内使用（MCP Client + 推理端 tool calling）**：如果部署了支持 MCP tool calling 的推理框架（vllm guided generation、Ollama tool call 模式等），模型的工具调用输出会在 **token 级别**被约束为合法格式。这对小模型而言是把软约束变成硬约束的最直接手段——比任何 prompt 技巧都更可靠。

## 六、总结性判断

| 技术/概念 | 本工程立场 | 核心理由 |
|-----------|-----------|---------|
| Tool（ToolRegistry）| 已实现，核心资产 | 确定性执行，小模型场景下首选扩充而非依赖 Cypher 生成 |
| Skill（Anthropic 标准）| 当前不适用 | 依赖强模型自主判断，与小模型场景和 LangGraph 控制流冲突 |
| PromptStrategy 类（当前 skills/）| 合理工程封装 | 把 prompt、校验、预判从节点代码剥离，提升可维护性；命名非 Anthropic 标准 |
| Harness Engineering | 核心设计原则 | LangGraph 图结构本身是 harness；节点间加校验检查点是该原则的直接落地 |
| MCP | 未来方向 | 推理端 tool calling 可把小模型的工具选择从软约束变硬约束；对外暴露能力的标准协议 |
| Agent Skills（开放标准）| 平台化时再考虑 | 当前内部工具场景价值有限；适合需要对外开放、第三方扩展能力时引入 |

**一句话**：在小模型场景下，**减少模型自主决策面** 是所有技术选型的首要标准。代码能决定的，不交给模型；模型能做的，只让它做最窄的那一步；软约束能换成硬约束的，优先换。
