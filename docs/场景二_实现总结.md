## 一、架构概览

```
Layer 1: 输入结构化 (Input Structuring)
  ┌────────────────────────────────────────────┐
  │  .docx → extract_outline → flatten → summary │
  │  输出: IntermediateLeaf[]                    │
  │  {id, path, heading, level,                  │
  │   original_content, summary, source}         │
  └────────────────────────────────────────────┘
                      ↓
Layer 2: 模板匹配填充 (Template Match & Fill)
  ┌────────────────────────────────────────────┐
  │  Phase 1: 三骨架对比 → GlobalGuidance       │
  │  Phase 2: 按 H1 分组并发 → 四态匹配         │
  │  Phase 3: 回写 docx + 来源追溯              │
  └────────────────────────────────────────────┘
```

### 设计原则

| 原则 | 说明 |
|------|------|
| **LLM 不看原文** | 正文超过 100 字做摘要，原文保留在内存，回写时代码直接取 |
| **章节级匹配** | 模板叶子与信息源叶子一对一匹配，上下文小，幻觉可控 |
| **四态填充** | skip(章标题跳过) / high(代码取原文) / partial(原文+LLM提炼) / missing(LLM生成) |
| **从尾到头回写** | 避免段落索引偏移，插入不影响前方未处理标题 |

## 三、核心 Skill 接口

### 3.1 SkeletonGuidanceSkill

```
输入:
  template_label: str        # 模板名称
  template_skeleton: str     # 模板骨架文本 (目录树)
  source_skeletons: [        # 信息源骨架列表
    {label: str, skeleton: str}, ...
  ]

输出:
  guidance: {
    terminology: str         # 术语统一表 (Markdown)
    scope_constraints: str   # 范围约束 (Markdown)
    fill_strategy: str       # 填充策略 (Markdown, 逐章标注来源+充足度)
  }

便捷方法:
  SkeletonGuidanceSkill.from_leaf_lists(
    template_label, template_leaves, source_leaf_map
  )
```

### 3.2 LeafMatchFillSkill

```
输入:
  template_leaves: [FlatLeaf, ...]   # 模板叶子列表
  source_leaves: [FlatLeaf, ...]     # 信息源叶子列表
  guidance: dict | None              # GlobalGuidance

输出:
  match_results: [{
    leaf_id: str
    match_level: "skip" | "high" | "partial" | "missing"
    source_refs: [str]              # 信息源叶子 ID
    supplement: str                 # partial 时填写
    generated_content: str          # missing 时填写
  }]
  stats: {total, skip, high, partial, missing}
```

匹配等级说明：

| 等级 | 触发条件 | 回写行为 |
|------|---------|---------|
| skip | 章标题(is_leaf=False 且无占位) | 不填内容 |
| high | 事实性章节有精确来源 | 代码取 original_content 直填 |
| partial | 总结性章节 / 信息不全 | 原文 + LLM 提炼/补充 |
| missing | 无信息来源 | LLM 生成 pure text |

### 3.3 DocxBackfill

```python
backfill_docx(
    template_path,          # 模板 .docx 路径
    output_path,            # 输出 .docx 路径
    match_results,          # Phase 2 匹配结果
    template_leaves,        # Phase 0 模板叶子
    source_leaves,          # Phase 0 信息源叶子
    default_body_style="FIT_WDT_正文",  # 正文样式, 可配置
)
# → {filled, skipped, traces, output}
```

## 四、数据流与 IntermediateLeaf

所有文档统一为 `IntermediateLeaf` 格式：

```python
{
    "id": "a3",                        # 唯一 ID (前缀: t=模板, a/b=信息源)
    "path": "需求管理 > 需求结构树管理 > PBS构建",
    "heading": "PBS构建方式",
    "level": 3,                         # 标题层级 (1-5)
    "original_content": "在产品初始化中...",  # 原文, 不给 LLM
    "summary": "支持基于PBS进行需求结构树定义...",  # ≤100字摘要, 给 LLM
    "is_leaf": True,                    # 是否叶子节点
    "parent_id": "a2",                  # 父节点 ID
    "source": "输入1-用户需求",          # 来源标识
}
```

`original_content` 永远不给 LLM，回写时由代码取。`summary` 给 LLM 做匹配判断。

## 五、接入 Plan Agent

### 5.1 自定义计划模板

在 `templates/` 目录下创建 `req_analysis_to_prd.md`：

```yaml
---
template_id: req_analysis_to_prd
short_description: 将需求分析报告+用户需求填充到PRD模板, 生成产品需求规格说明书
applicable: 用户有结构化的需求文档和需求分析报告, 需要生成PRD
expected_output: 填充后的产品需求规格说明书 .docx
---

# 模板: 需求分析 → PRD

## 步骤

1. file_parse
   reason: 解析输入文档为中间格式(叶子平铺+摘要)
   params:
     file_index: <用户提供的输入文档路径>
     mode: leaf_flatten
     heading_styles: "Heading,FAI"
     summary_threshold: 100

2. file_parse
   reason: 解析PRD模板为骨架
   params:
     file_index: <PRD模板路径>
     mode: skeleton_only

3. skeleton_guidance
   reason: 生成全局填充指导(术语/范围/策略)
   depends_on: [1, 2]

4. leaf_match_fill
   reason: 叶子级匹配填充
   depends_on: [1, 2, 3]

5. export_data
   reason: 回写docx
   depends_on: [4]
   params:
     format: docx
     body_style: "FIT_WDT_正文"
```

### 5.2 Registered Capabilities

场景二用到的 Capability 均已在 `CapabilityRegistry` 自动注册：

| 能力名 | 类别 | 用途 |
|--------|------|------|
| `skeleton_guidance` | skill | 多骨架对比全局指导 |
| `leaf_match_fill` | skill | 叶子四态匹配填充 (skip/high/partial/missing) |
| `file_parse` | skill | 文档解析(已有, 新增 leaf_flatten 模式) |

### 5.3 AutonomousAgent 编排流程

```
用户上传: 输入1.docx + 输入2.docx + 模板.docx
     ↓
TemplateRecall: 匹配模板 req_analysis_to_prd
     ↓
PlannerSkill: 生成 Plan (5 步)
     ↓
_execute_plan():
  Step 1: FileParseSkill → IntermediateLeaf[] (输入1)
  Step 2: FileParseSkill → IntermediateLeaf[] (输入2)
  Step 3: FileParseSkill → 模板骨架
  Step 4: SkeletonGuidanceSkill → GlobalGuidance
  Step 5: LeafMatchFillSkill → MatchResult[]
  Step 6: ExportSkill → 回写 docx
     ↓
_summarize_results() → 用户看到输出
```

## 六、扩展新场景

### 6.1 需要做的事

1. **准备输入文档** — 提供信息源 docx + 模板 docx
2. **写计划模板** — `templates/xxx.md`, 定义步骤和参数
3. **调参** — `heading_styles` (标题样式匹配), `summary_threshold` (摘要阈值)
4. **验证** — `python test/e2e_scenario2.py --phase 0` 看叶子平铺效果

### 6.2 可配置参数

| 参数 | 位置 | 默认值 | 说明 |
|------|------|--------|------|
| `heading_styles` | `flatten_document()` | `("Heading",)` | 标题样式前缀, 如 `("Heading","FAI")` |
| `summary_threshold` | `flatten_document()` | `100` | 超过此字数触发 LLM 摘要 |
| `summary_concurrency` | `flatten_document()` | `5` | 摘要并发数 |
| `id_prefix` | `flatten_document()` | `"s"` | 叶子 ID 前缀, 防多文档 ID 碰撞 |
| `default_body_style` | `backfill_docx()` | `""` | 输出 docx 正文样式名 |
| `_MATCH_CONCURRENCY` | `leaf_match_fill.py` | `3` | 匹配 LLM 并发组数 |

### 6.3 场景差异处理

不同领域的差异只需改两处：

| 差异点 | 配置方式 |
|--------|---------|
| 文档标题样式不同 | `heading_styles` 参数 |
| 正文样式名不同 | `default_body_style` 参数 |
| 章节结构不同 | 自动适配 (extract_outline 动态根层级) |

## 七、已知限制与后续方向

| 限制 | 影响 | 方向 |
|------|------|------|
| 模板 docx 样式需预定义 | `default_body_style` 指向的样式必须在模板中存在 | 回写时自动创建样式 |
| 总结性章节识别靠 prompt 规则 | 非"现状/思路/分析"类标题不走提炼 | Phase 1 手动标注总结性章节 |
| 正文回写为纯文本段落 | 模板中如有表格/图片, 会被清除 | FAQI 表格填充支持 |
| TOC 直接被删除 | 需要用户在 Word 中手动插入目录 | 回写后自动插入 TOC 域 |
| `extract_outline` 只处理 body 段落 | 页眉/页脚/文本框不提取 | 按需扩展 |
